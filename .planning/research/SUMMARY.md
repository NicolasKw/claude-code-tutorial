# Project Research Summary

**Project:** Claude Code Mastery — Tutorial Interactivo (Lead Magnet)
**Domain:** Interactive tutorial lead magnet with LinkedIn OAuth, progress tracking, and viral certificate sharing
**Researched:** 2026-04-02
**Confidence:** HIGH

## Executive Summary

This is a single-path, ~1-hour interactive tutorial targeting non-technical Spanish-speaking business owners who arrive from a LinkedIn post. The product is a lead magnet: its job is to capture a LinkedIn lead, deliver real learning value (the user builds a working personal management bot using Claude Code on their own machine), and trigger a LinkedIn share via a personalized certificate. The stack is straightforward — Next.js 15 App Router on Vercel, Auth.js v5 with LinkedIn OIDC, Neon + Drizzle for progress persistence, and `next/og` for server-side badge generation. All chosen technologies have well-documented patterns and HIGH confidence from official sources.

The recommended approach is a multi-page server-rendered app (not a SPA) where each of the 7 tutorial levels has a real URL, auth gating happens server-side, and tutorial content lives in static TypeScript files (not the DB). The data model is intentionally minimal: two tables (users, progress). The certificate is a stateless on-demand PNG generated by `next/og`, served from `/api/og?userId=X`, with a shareable public page at `/certificate/[userId]` that includes correct Open Graph meta tags so LinkedIn auto-previews it.

The single biggest risk is timeline, not technical: LinkedIn Developer App approval for the OIDC product takes 1-3 business days and is a hard blocker for any auth work. This must be started on day one. The second risk is UX abandonment — non-technical users will drop if the tutorial lacks clear action checkpoints, copy-paste code blocks, and error fallbacks. The third risk is viral loop failure — the certificate must look professional and have a stable public URL, or users will not share it.

## Key Findings

### Recommended Stack

Next.js 15 App Router is the clear choice: it collapses backend API routes, server components, and `next/og` certificate generation into one framework with zero-config Vercel deployment. Auth.js v5 handles LinkedIn OIDC with the correct scope set (`openid profile email`) — note that the legacy LinkedIn v2 OAuth is deprecated and must not be used. Neon (serverless Postgres) on the free tier is sufficient for thousands of users given the minimal data model; Drizzle ORM keeps the schema type-safe without heavy overhead. If the developer already has a Supabase account, that is an equally valid substitute for Neon.

**Core technologies:**
- Next.js 15 App Router: framework — collapses frontend, API routes, and `next/og` into one deployment
- Auth.js v5 + LinkedIn OIDC: authentication — correct approach for LinkedIn sign-in in 2026; legacy v2 OAuth is deprecated
- Neon + Drizzle ORM: database — serverless Postgres free tier sufficient; type-safe schema
- Tailwind CSS v4: styling — zero config, fast iteration, v4 stable and removes `tailwind.config.js`
- `next/og` (ImageResponse): certificate generation — built-in, no extra install, CDN-cached, og:image-compatible
- Vercel: hosting — zero config with Next.js, Edge Runtime for `next/og`, Neon Marketplace integration

### Expected Features

**Must have (table stakes):**
- LinkedIn OAuth registration gate — 1-click, captures lead before content access
- Clear value proposition on landing BEFORE the login CTA — show what they build, time estimate, social proof
- Linear 7-level progression with visual unlock feedback — one level visible at a time
- Progress persistence server-side (DB, not localStorage) — cross-device, survives tab close
- Copy-paste code blocks with 1-click copy — non-technical users cannot retype commands
- Explicit action checkpoints per step — "Listo, ya lo hice" button; no ambiguity about when to act
- Error fallback callouts per step — non-technical users WILL hit errors; without fallback they abandon
- Personalization with LinkedIn name — "Hola [nombre]" in header and certificate

**Should have (differentiators):**
- Personalized downloadable PNG certificate — the core viral trigger; must look professional
- Pre-written LinkedIn post text in an editable textarea — eliminates friction between intent and action
- LinkedIn share button (`share-offsite` URL, no API needed) — 1-click viral loop
- "What you built" summary screen on completion — closes the learning loop
- Time estimate per level ("~8 min") — reduces commitment anxiety for busy business owners
- Fast-path collapsibles for users who already have tools installed
- Social proof counter on landing — "X personas ya completaron esto"

**Defer (v2+):**
- Mid-tutorial share CTAs
- Creator analytics dashboard
- Gamification beyond level unlock feedback
- Dark mode

### Architecture Approach

The app is a multi-page server-rendered Next.js app — not a SPA. Auth gating is enforced by Next.js middleware at the server level (`/tutorial/*` routes redirect unauthenticated users to `/`). Tutorial content lives as static TypeScript files in the repo (`content/levels/level-N.ts`), not in the DB — this makes content updates a code deploy, not a DB migration. The data model has exactly two tables: `users` (LinkedIn sub, name, email nullable, picture URL) and `progress` (completed_levels as integer array, last_seen_level). Sessions use JWT stored in an httpOnly cookie — no sessions table needed.

**Major components:**
1. Auth Layer — Auth.js v5 middleware + LinkedIn OIDC + user upsert on first login
2. Tutorial Content Layer — static TS files per level; renders server-side with progress state from DB
3. Progress Layer — `POST /api/progress` updates DB on level completion; server reads progress on each page load
4. Badge / Certificate Layer — stateless `next/og` PNG at `/api/og?userId=X`; public certificate page at `/certificate/[userId]` with OG meta tags
5. Landing / Registration — public `/` with value prop + LinkedIn login CTA; redirects to current level on return visits

### Critical Pitfalls

1. **LinkedIn OIDC product requires manual approval (1-3 days)** — create the LinkedIn Developer App and request the OIDC product on day one; use a mock OAuth for local development while waiting; do not let other phases block on this
2. **LinkedIn Developer App must be moved to Production Mode before real users can log in** — Development Mode only allows listed testers; include this in the pre-launch checklist with buffer days
3. **Redirect URI must match exactly (including trailing slash)** — register the exact URI in LinkedIn Developer Portal and replicate it in code; document in `.env.example`; mismatch causes silent OAuth failure
4. **LinkedIn profile photo URLs expire (days to weeks)** — do NOT store the photo URL for use in the badge; for v1, use name only in the certificate design (simpler and reliable)
5. **LinkedIn email field is optional in OIDC response** — make `email` nullable in the DB schema and handle `null` in the Auth.js callback; a non-nullable email column will cause 500 errors on sign-up for some users

## Implications for Roadmap

### Phase 1: Foundation + Auth
**Rationale:** Everything else requires knowing who the user is. LinkedIn OAuth approval is on the critical path and has a 1-3 day external dependency — start this phase first to absorb the wait time.
**Delivers:** Working LinkedIn sign-in, user record created in DB, session established, middleware protecting `/tutorial/*`
**Addresses:** Registration gate (table stakes), lead capture (core value)
**Avoids:** Redirect URI mismatch, nullable email crash, Development Mode user lockout

### Phase 2: Progress Infrastructure + Tutorial Shell
**Rationale:** Auth is in place; now establish the skeleton that content will fill. Progress API must exist before any level can be "completed." The tutorial shell (layout, level unlock logic, navigation) is content-independent and faster to build than the content itself.
**Delivers:** `/tutorial/[level]` routes rendering static content, level unlock logic, progress API (`POST /api/progress`), progress persistence in DB
**Addresses:** Persistent progress (table stakes), linear progression (table stakes)
**Avoids:** localStorage-only progress (cross-device failure)

### Phase 3: Tutorial Content (7 Levels)
**Rationale:** The shell is ready; fill it. This is the highest-effort phase and the core product value. Content must be authored carefully for a non-technical audience — action-first, copy-paste code blocks, error fallbacks.
**Delivers:** All 7 levels with step-by-step instructions, code blocks, action checkpoints, error callouts
**Addresses:** Copy-paste blocks, action checkpoints, error fallbacks, fast-paths for experienced users, time estimates per level
**Avoids:** Prerequisite overload consuming the 1-hour budget (communicate prerequisites on landing and in Level 0)

### Phase 4: Certificate + Sharing Mechanics
**Rationale:** Certificate and sharing are the viral trigger — build them after the tutorial works end-to-end. Requires user data from DB (established in Phase 1) and a working tutorial to complete (Phase 3).
**Delivers:** `/api/og` PNG generation, `/certificate/[userId]` public page with OG meta tags, download button, LinkedIn share URL, pre-written post text textarea
**Addresses:** Downloadable certificate (differentiator), LinkedIn share (differentiator), completion summary screen
**Avoids:** Client-side Canvas badge (CORS taint error), missing public certificate URL (breaks og:image preview), using LinkedIn UGC API (requires Partner Program — use share URL instead)

### Phase 5: Landing Page + Polish
**Rationale:** Conversion-optimized entry point built last when the full product is known. Social proof counter requires real users (Phase 1-4 complete). Polish depends on seeing the full flow.
**Delivers:** Production landing page (value prop, time estimate, creator presence, social proof counter), responsive mobile layout, personalization ("Hola [nombre]"), visual unlock animations
**Addresses:** Value proposition before registration gate (critical pitfall #5), social proof counter, creator branding
**Avoids:** High bounce rate from cold LinkedIn traffic hitting a bare login button

### Phase 6: Pre-Launch Hardening
**Rationale:** Several LinkedIn-specific steps must happen before real users can access the app. Buffer time needed for external approvals.
**Delivers:** LinkedIn app moved to Production Mode, all redirect URIs verified, scopes finalized (do not change post-launch), `.env.example` documented, smoke test with a real LinkedIn account
**Avoids:** Development Mode user lockout, post-launch scope change invalidating all tokens

### Phase Ordering Rationale

- Auth first because LinkedIn approval is an external dependency with a multi-day wait — start it before writing a single component
- Progress infrastructure before content because the shell must exist before content can be loaded into it
- Certificate last among features because it requires a complete user record and a completable tutorial
- Landing last because it references real numbers (social proof) and the actual product
- Hardening as a dedicated phase because LinkedIn Production Mode approval is another external-dependency step that needs its own buffer

### Research Flags

Phases with standard patterns (skip research-phase):
- **Phase 1 (Auth):** Auth.js v5 + LinkedIn OIDC is well-documented; official Microsoft/LinkedIn docs verified
- **Phase 2 (Progress):** Simple Postgres read/write patterns; no novel architecture
- **Phase 4 (Certificate):** `next/og` is well-documented by Vercel; share URL pattern is standard
- **Phase 5 (Landing):** Standard conversion-optimized landing; no novel patterns

Phases that may benefit from deeper research during planning:
- **Phase 3 (Tutorial Content):** The pedagogical structure (how to sequence the 7 Claude Code levels into ~1-hour steps for non-technical users) is a content design problem, not a tech problem — may need iteration with the creator

### Gaps to Address

- **Tutorial content structure:** The 7 levels need to be authored from scratch using the LinkedIn post and YouTube video transcript as source material. The pedagogical sequencing (which concepts go in which level, how many steps per level to stay within ~1 hour) is the main open design question and should be validated with the creator before Phase 3 begins.
- **Certificate visual design:** The badge design needs to look professional enough to share. This is a design decision (not a tech decision) that affects whether the viral loop works. Should be prototyped before full implementation in Phase 4.
- **Prerequisites communication:** The exact list of prerequisites (Node.js? Git? Claude Code CLI? API key?) and their setup time needs to be pinned before content is authored — it directly affects whether the 1-hour budget is achievable.
